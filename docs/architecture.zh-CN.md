# MemWeave（忆织）架构说明

这份文档记录修改 MemWeave 时不能打破的数据关系。核心原则很简单：原始资料要保留，对资料的解释要能找回来源，聊天所用的产物要能随记忆变化重新生成。

MemWeave 不是身份替代系统。聊天和声音输出不能声称自己是真实本人、具有意识或已被“复活”，也不能把没有证据的内容说成事实。

## 数据主线

```text
raw_sources
    -> source_segments
        -> persona_items（A-M）
            -> generated Skill versions
                -> selected runtime slices
                    -> chat reply
```

- `raw_sources` 保留用户提交的材料和提取内容。
- `source_segments` 标出真正需要使用的片段，并记录它属于谁。
- `persona_items` 保存关联来源的 A-M 解释，同时记录置信度、稳定性、适用范围和风险。
- `uncertain_items` 和 `question_targets` 暂存不清楚、有冲突、较敏感或低置信度的解释，等待用户处理。
- 生成的 Skill 把已审核内容组装成运行时产物，记忆改变后可以重建。
- `chat_records` 只保存对话历史。聊天里的新说法在用户确认前仍是候选。

## A-M 人格记忆库

| 分组 | 内容 |
| --- | --- |
| A | 事实与经历 |
| B | 语言风格 |
| C | 情绪反应 |
| D | 性格特征 |
| E | 价值观与世界观 |
| F | 关系模式 |
| G | 决策逻辑 |
| H | 冲突与防御 |
| I | 关心与陪伴 |
| J | 场景反应 |
| K | 成长与变化 |
| L | 边界与置信度 |
| M | 声音与说话特征 |

## 后端

FastAPI 后端把 HTTP 处理、存储、单域规则和跨域流程拆开。

- API 路由校验请求并暴露 `/api/v1` 资源。
- Store 一次只管理一类记录。
- Workflow 协调摄取、问题回答、不确定项处理、Skill 生成和聊天候选等跨数据操作。
- 运行时安全门把隐藏、遗忘、无证据或归属错误的内容挡在聊天和 Skill 之外。

默认存储是本地 JSON。可选 SQLite 模式保存同样的 JSON 文档，它不是面向多用户的规范化数据库设计。

## 前端

Next.js 工作台围绕五个用户任务组织：

- Dashboard：创建/选择人物并添加原始资料。
- Library：检查证据、Persona 条目、不确定项和 Skill 版本。
- Chat：检索有限记忆切片，并可选生成授权声音。
- Interview：先把访谈答案保存成 raw source，再进入分类。
- Settings：配置模型供应商、功能开关、存储与诊断。

旧 Translate 页面不在当前公开主线中。

## 外部模型

模型调用使用兼容 OpenAI 协议的接口。聊天/分类、视觉 OCR 和 ASR 可以分别配置 Base URL、API Key 和模型名。模型发现只用来帮助选择；供应商不支持发现时，仍可以手动输入模型名。

外部 AI 默认关闭。提取或分类失败时，raw source 仍然保留；本地规则也不能被报告成正式模型蒸馏成功。

## 声音输出

M 组保存的是声音和说话特征描述，这一步本身不是声音克隆。可选 IndexTTS2 是独立的后续步骤，必须有授权参考、确认的来源归属、明确的本次生成同意和固定 `reply_text`。模型权重与音频资产不进入本仓库。

## 公网部署前

当前后端只按本地单用户设计。公网部署前至少需要补齐鉴权、多租户隔离、生产存储、上传限制、TLS、密钥管理和运维监控。
